### <u>Уровень 1</u>
### Основной функционал приложения Mesto:
Я как пользователь могу:
1. добавлять/удалять фотографии мест (карточка) с их названием
2. добавлять/удалять лайки на карточки
3. редактировать свой профиль (фото, имя, описание)
4. для входа в приложение просмотра карточек, мне необходимо пройти аутентификацию/регистрацию

### Список микрофронтов
Проект можно разбить по принципу *Вертикальноей нарезки (DDD)* на следующие микрофронты:
#### 1. `auth_microfrontent`
Инкапсулирует в себе логику аутентификации, авторизации и регистрации новых пользователей в приложении.
#### 2. `card_microfrontent`
Инкапсулирует в себе логику по работы с карточками: 
- добавление / удаление карточек
- добавление / удаление лайков
#### 3. `user_microfrontent`
Инкапсулирует в себе логику по работы по созданию профиля и его редактирования
#### 4. `host_microfrontent`
Точка входа в приложение, содержит общие для всех микрофронтов компоненты и динамически подгружает компоненты из разных микрофронтов


### Фреймворк
В качестве фреймоврка для создания микрофронтентов можно использовать `Webpack Module Federation` по следующим причинам:
1. Все компоненты написаны на React, хост и удалённые модули могут использовать общие зависимости
2. В курсе все примеры приведены для Module Federation, будем считать, что у гипотетической команды есть опыт работы именно с этим фреймворком

### Управление состоянием
Можно использовать библиотеку глобального состояния, т.к в проекте уже используется контекст `CurrentUserContext`. Это состояние используется несколькими микрофронтами

# Задание 2
 *Декомпозировать веб-приложения на Django на микросервисы*

### <u>Бизнес-процессы текущего решения</u>
По предоставленной информации, с определенной долей вероятности и допущениями могу предположить, что данное веб-приложение (портал) для управления торговыми площадками имеет следующий функционал:
1. Пользователя могут размещать свои товары и услуги, выступая в качестве `продавца`, а также выступать в качестве `покупателя`, размещая заказы на оказание услуг и покупки товаров
2. Сделки между `покупателями` и `продавцами` могут заключаться как на аукционах, так и по прямому контракту
3. Портал интегрируется с внешними платежными системами для совершения оплат на торговых площадках
4. Пользователи получают сообщения о различных событиях на площадках
5. Пользователи могут запросить различные отчеты о совершенных операциях на площадках
6. В случае спорных ситуаций, пользователи могут обратится за апелляцией (арбитраж) к администраторам площадки, в случае технических вопросов - к специалистам технической поддержки портала 
7. Портал имеет систему аутентификации и авторизации, выделяя 3 основные роли:
	- **клиент**: *управляет только своими данными, реагирует заказы других пользователей*
	- **администратор**: *регистрация новых пользователей площадок, арбитраж и модерация аукционов*
	- **специалист технической поддержки**: *обработка заявок на техподдержку от пользователей и администраторов*

### <u>Разделение монолита на компоненты</u>
Монолит можно условно разделить на следующие компоненты:
1. **Пользователь**.
> Отвечает за управление данными о пользователях (`клиент`, `администратор`, `специалист технической поддержки`), регистрацию, аутентификацию и авторизацию и ролевую модель. 
> *Также можно было бы вывести функции связанные с аутентификации, авторизации в отдельный авторизационный сервис или использовать готовые решения типа keycloack и т.д.*
2. **API Gateway**
> Маршрутизация и формато-логический контроль запросов, аутентификация (проверка JWT токена)
3. **Услуга**
> Отвечает за управление данными о предоставляемых пользователями услугами
4. **Заказ**
> Отвечает за управление данными о заказах пользователей.
5. **Товар**
> Отвечает за управление данными о товарах пользователей.
6. **Аукцион**
> Инкапсулирует в себе логику по работе с аукционами на площадках
7. **Апелляция**
> Инкапсулирует в себе логику по работе с апелляциями пользователей
8. **Поддержка**
> Отвечает за управление данными о заявках на техподдержку от пользователей.
9. **Сервис нотификации**
> Слушает определенные топики брокера сообщений и на их основе отправляет сообщения во внешний сервис нотификаций.
10. **Внешний сервис нотификации**
> Отправляет нотификации пользователям/администраторам/специалистам техподдержки портала по необходимым каналам.
11. **Отчеты**
> Предоставляет агрегированные отчеты и статистику по операциям пользователя. Запрашивает необходимые данные у соответствующих микросервисов. 
> *Межсервисное взаимодействие и взаимодействие с пользователем может быть асинхронным через брокер сообщений и уведомление пользователя о его готовности, если построение отчета будет занимать значительное время и ресурсы БД*
12. **Торговая площадка**
>  Отвечает за проведение финансовых операций по совершенным сделкам, взаимодействует с внешними платежными системами
13. **Брокер сообщений**
>  Необходим для асинхронного взаимодействия сервисов. Отправка нотификаций, статусы платежей по заказам, отчеты. 
>  *в качестве реализации можно использовать Apache Kafka*
14. **Портал электронной площадки**
> Клиентское приложение, через него пользователь взаимодействует с торговыми площадками
15. **Портал администратора**
> "админка" веб-приложение для работы `администраторов` площадки
16. **Портал технической поддержки**
> веб-приложение для работы `специалистов технической поддержки` площадки
17. **BFF для портала электронной площадки**
> bff для клиентского веб-приложения инкапсулирует в себе все необходимые внутренние запросы к микросервисам и подготавливает входные/выходные данные в нужном формате
18. **BFF для внутренних порталов**
> bff для веб-приложениий администраторов и техподдержки инкапсулирует в себе все необходимые внутренние запросы к микросервисам и подготавливает входные/выходные данные в нужном формате. 
> *можно разделить на 2 отдельных сервиса, свой под каждый внутренний портал*

1. пользователь создает заявку на создание/редактирование аукциона/апелляции/техподдержки
2. администратор/сотрудник техподдержки получает уведомление по электронной почте
3. администратор/сотрудник техподдержки выполняет действие по заявке
4. клиент получает уведомление о об изменении статуса заявки

Также, при необходимости, сервисы могут напрямую обращаться друг к другу для "обогащения" данных, на схеме данные потоки не отображены
